    CAS Workflow Execution Model

    Introduction

    The CAS Workflow Execution Model defines how a workflow becomes a first-class executable capability within the Continuous Architecture System (CAS).

    This model formalises the execution contract already present in CAS Runner and the Knowledge Layer. It does not redesign CAS Runner. It explains how current workflow execution works, where responsibilities sit, and what onboarding conditions future workflows must satisfy before they can be treated as runtime-compatible CAS capabilities.

    CAS already contains governance models that explain why governance exists, how projects enter governance, and how workflows participate in Prevent, Detect, and Feedback loops. This model explains the execution boundary beneath those models:

    • CAS Runner orchestrates.
    • Workflow steps prepare, reason, validate, and return governed outputs.
    • The Reasoning Execution Contract mediates semantic reasoning.
    • The Knowledge Provider supplies and persists governed knowledge.
    • Prompts and schemas act as executable workflow contracts.

    The model is grounded in the current CAS implementation, including the registered runtime flows, the step execution pattern, prompt discovery, schema validation, provider abstraction, telemetry, and Knowledge Provider I/O boundary.

    Workflow Philosophy

    CAS workflows are operational governance capabilities, not isolated prompts.

    A first-class workflow must be executable, bounded, observable, and able to participate in the CAS knowledge lifecycle. It must convert governed inputs into governed outputs without taking ownership of orchestration concerns that belong to the runner.

    Runner Orchestrates

    CAS Runner owns execution sequencing, shared context, provider initialization, configuration resolution, telemetry, evaluation, and persistence coordination.

    The runner decides which steps execute and in what order. Workflow steps do not self-schedule or decide the pipeline topology at runtime.

    Workflow Reasons

    Workflow steps own domain transformation. A step prepares the inputs it needs, invokes deterministic logic and semantic reasoning where appropriate, validates its output shape, and returns named outputs to the runner.

    Workflow steps should not own global flow orchestration, persistence routing, provider selection, or cross-step scheduling.

    Knowledge Provider Provides Knowledge

    The Knowledge Provider owns the persistence and retrieval boundary for governed knowledge. Workflows consume knowledge through this boundary instead of directly inventing storage rules.

    The current local provider loads rules, project vision, architecture work, latest AID, foundational assets, project configuration, and capability lineage. It persists known output types such as rule context, architecture work, AEM, alignment reports, AID, baseline assessments, execution profiles, capability lineage, and evaluations.

    Prompts Are Contracts

    Prompt files are not incidental text. In CAS Runner they are loaded through the Prompt Repository as versioned reasoning definitions. Prompt metadata can name schemas and semantic contract versions. The Reasoning Execution Contract injects runtime input context, invokes the provider, and validates structured output where a schema exists.

    Deterministic Core, Selective Augmentation

    CAS uses deterministic orchestration and deterministic base computations where possible, with LLM reasoning bounded behind provider and reasoning contracts. Alignment, AEM, AID, discovery, and rule-context steps may use semantic reasoning, but deterministic summaries, fallback paths, schema validation, and explicit output keys preserve execution integrity.

    Execution Model Overview

    At runtime, CAS executes a workflow as a fixed, dependency-ordered pipeline:

    Flow request
     |
     v
Resolve registered flow
     |
     v
Initialize execution context
     |
     v
Load project knowledge inputs
     |
     v
For each registered step:
  prepare_inputs(context)
  execute(inputs, reasoning_engine)
  validate(output)
  persist_outputs(output)
  update shared context
     |
     v
Persist telemetry and run evaluation

    This implements the orchestrator-managed-pipeline pattern. The runner provides a predictable execution sequence, explicit step handoffs, and traceable data lineage across workflow stages.

    The current runtime registry includes:

    Flow Current registered steps Execution role
    orchestration-flow rules-context-loader, architecture-work-to-evolution-map, alignment-engine, architecture-intent-generator Normal governance orchestration loop
    discovery-flow baseline-discovery, rules-context-loader, architecture-work-to-evolution-map, alignment-engine, architecture-intent-generator Baseline discovery followed by governance state generation
    steering-projection-flow steering-projection Steering drift and projection workflow

    The conversation-to-architecture-work step exists in code and prompt/workflow documentation, but it is not currently active in the registered orchestration flow because it is commented out in the flow registry. It should therefore be treated as implemented but not currently onboarded as an active runtime step in the default flow.

    Workflow Lifecycle

    A CAS workflow becomes executable through a lifecycle that moves from governed context to persisted outputs.

    1. Context Acquisition

    The runner initializes project-level execution context.

    Current context includes:

    • project identifier
    • project root
    • workspace root
    • rule context
    • architecture work
    • latest AID
    • project vision
    • patterns
    • project configuration
    • resolved execution profile
    • observability provider

    Before step execution, the runner loads current architecture work, latest AID, and project vision from the Knowledge Provider. Individual steps can prepare additional inputs, such as global and project rules or foundational assets.

    2. Dependency Resolution

    Dependencies are satisfied through explicit step ordering and shared context.

    Each step receives the current context and prepares only the input shape it needs. Outputs from prior steps are merged back into context and become available to downstream steps.

    Current examples:

    • rules-context-loader loads global and project rules and returns rule_context.
    • architecture-work-to-evolution-map consumes architecture work, baseline assessment, rule context, vision, project, and date.
    • alignment-engine consumes AEM, architecture work, rule context, AID or baseline assessment, project, date, flow name, and prompt base path.
    • architecture-intent-generator consumes AEM, baseline assessment, alignment report, vision, project, and date.
    • baseline-discovery consumes foundational assets and can emit both a baseline assessment and proposed architecture work.

    3. Reasoning Execution

    Semantic reasoning is invoked through the Reasoning Execution Contract, not directly through arbitrary provider calls.

    The contract:

    • identifies the active flow and step
    • loads the latest matching prompt definition
    • supports fallback to shared orchestration prompts
    • parses prompt frontmatter metadata
    • injects runtime input context into the prompt
    • invokes the configured provider
    • loads and applies JSON schemas where available
    • validates structured responses
    • emits telemetry attributes and validation events

    This realizes the prompt-as-code-separation, structured-output-via-schema, and llm-provider-abstraction-boundary patterns.

    4. Output Validation

    Each step validates the output key and the minimum expected structure before the runner persists or propagates results.

    Examples:

    • rules-context-loader requires rule_context.
    • architecture-work-to-evolution-map requires aem.frontmatter and aem.sections.
    • alignment-engine requires alignment_report.frontmatter and alignment_report.sections.
    • architecture-intent-generator requires aid.frontmatter and aid.sections.
    • baseline-discovery requires baseline_assessment.

    Schema validation also occurs inside the Reasoning Execution Contract when a JSON schema is available for the step.

    5. Persistence

    The runner delegates persistence to the Knowledge Provider through persist_outputs.

    Persistence is output-key driven. The current provider knows how to persist defined artifact types to governed locations in the knowledge workspace. Workflows should therefore emit recognized output keys or require explicit provider support before onboarding.

    6. Context Propagation

    After persistence, the runner updates shared context with the step output.

    List outputs are extended. Non-list outputs are replaced or added by key. This makes step outputs available to later steps without requiring downstream steps to reload the same artifact from disk.

    7. Telemetry and Evaluation

    Non-dry-run executions persist trace data to the telemetry root and run the evaluation harness against the captured trace.

    This makes workflow execution observable and supports future operational intelligence about reasoning quality, provider behavior, cost, and execution reliability.

    Responsibility Boundaries

    CAS Runner Responsibilities

    CAS Runner owns:

    • flow selection
    • registered step sequencing
    • execution context initialization
    • configuration resolution
    • LLM provider initialization
    • Reasoning Execution Contract initialization
    • project knowledge preloading
    • invocation of step lifecycle methods
    • persistence coordination
    • context propagation
    • telemetry persistence
    • orchestration evaluation

    The runner should remain deterministic in its core orchestration behavior.

    Workflow Step Responsibilities

    Workflow steps own:

    • declaring a stable step name
    • preparing step-specific inputs from context
    • applying deterministic domain logic where appropriate
    • invoking semantic reasoning through the reasoning contract when needed
    • producing named output keys
    • validating required output structure
    • preserving clear boundaries between deterministic logic and semantic augmentation

    Workflow steps should not bypass the Knowledge Provider for governed persistence or directly select LLM providers.

    Prompt Responsibilities

    Prompts own:

    • semantic task framing
    • transformation rules
    • output expectations
    • schema references where applicable
    • semantic contract metadata where applicable

    Prompts should not define runner orchestration policy or persistence topology.

    Schema Responsibilities

    Schemas own:

    • machine-readable output contracts
    • validation of structured reasoning responses
    • stability of downstream artifact shape

    Any workflow that produces structured knowledge intended for persistence should have an explicit schema or a documented reason why schema validation is not yet available.

    Knowledge Provider Responsibilities

    The Knowledge Provider owns:

    • loading governed inputs from the knowledge workspace
    • resolving the active knowledge workspace root
    • loading project configuration from the governed workspace
    • exposing a stable persistence/retrieval abstraction
    • persisting recognized output types to governed locations
    • preserving backward-compatible functional APIs where current flows still depend on them

    The provider boundary allows future storage implementations without changing workflow reasoning semantics.

    Execution Contract

    A workflow becomes first-class in CAS Runner when it satisfies the following execution contract.

    Flow Contract

    A flow must have:

    • a stable flow name
    • an ordered list of registered step names
    • a clear lifecycle purpose
    • known upstream trigger conditions
    • known output artifacts
    • a relationship to the project lifecycle or governance participation loop

    Future Workflow Manifest Alignment

    The Flow Contract defines the minimum governance information required for a first-class workflow. A future Workflow Manifest may become the machine-readable representation of this contract.

    If introduced, a manifest should describe workflow identity, lifecycle phase, category, inputs, outputs, prompts, schemas, dependencies, and onboarding status without changing the underlying execution architecture.

    The manifest should be treated as governance metadata rather than executable orchestration logic. CAS Runner remains responsible for orchestration, while manifests improve governance, discoverability, onboarding, and operational visibility.

    Step Contract

    A step must provide:

    • name
    • prepare_inputs(context)
    • execute(inputs, reasoning_engine)
    • validate(output)

    The step name is the join point between:

    • the flow registry
    • prompt discovery
    • schema mapping
    • telemetry naming
    • output validation

    Prompt Contract

    If a step uses reasoning, it must have an available prompt definition.

    Prompt discovery currently checks:

    1. latest versioned prompt under the active flow path
    2. unversioned prompt under the active flow path
    3. latest versioned prompt under orchestration-flow as a shared fallback
    4. unversioned prompt under orchestration-flow as a shared fallback

    Missing prompts result in an empty reasoning response rather than arbitrary unmanaged behavior.

    Output Contract

    A step output must:

    • be a dictionary
    • use stable top-level output keys
    • satisfy step-level validation
    • satisfy schema validation where a schema applies
    • be persistable by the Knowledge Provider if it represents governed knowledge
    • be safe to merge into shared context for downstream steps

    Observability Contract

    Reasoning steps should be traceable through telemetry attributes including provider, model, temperature, flow name, step name, prompt path, prompt version, schema, semantic contract version, token usage, and estimated cost when available.

    Dependency Contract

    CAS workflow dependencies are explicit, not hidden.

    Configuration Dependencies

    Runtime configuration must be resolved through cas-config.yaml, environment variables, CLI overrides, and defaults according to the runner configuration loader.

    Changes to provider settings or gateway behavior must be reviewed for stability and operational cost impact.

    Knowledge Dependencies

    Knowledge dependencies must be loaded through the Knowledge Provider or through runner-provided context.

    Examples:

    • rules
    • architecture work
    • AID
    • vision
    • foundational assets
    • project configuration
    • capability lineage

    Workflows should not depend on legacy configuration files or deprecated knowledge paths.

    Prompt Dependencies

    Prompt dependencies must be stored in the configured prompt base path and resolved by the Prompt Repository.

    Versioned prompts should be preferred when a workflow is expected to evolve while preserving compatibility.

    Schema Dependencies

    Structured workflow outputs should be backed by schemas under the runner schema set or by another explicit schema location referenced by prompt metadata.

    Step Dependencies

    Step dependencies are currently represented by order in the flow registry and by named context keys. A downstream step depends on upstream outputs by reading the corresponding context keys during prepare_inputs.

    This is sufficient for the current fixed pipelines. Future dynamic or branching flows would require an explicit dependency graph rather than implicit ordering alone.

    Workflow Categories

    CAS workflows can be grouped by their governance role.

    Discovery Workflows

    Discovery workflows establish understanding before governance reasoning proceeds.

    Current example:

    • discovery-flow
    • baseline-discovery

    These align with the Baseline Discovery phase of the CAS Project Lifecycle Model.

    Governance Workflows

    Governance workflows participate in Prevent, Detect, and Feedback loops by loading rules, evaluating alignment, producing architecture work, and generating governance intelligence.

    Current examples:

    • rules-context-loader
    • alignment-engine
    • steering-projection-flow

    These align with the Governance Workflow Mapping and Operational Governance Model.

    Intelligence Workflows

    Intelligence workflows synthesize state, intent, drift, and capability signals into higher-level governance understanding.

    Current examples:

    • architecture-work-to-evolution-map
    • architecture-intent-generator
    • steering projection generation

    These support Continuous Governance by making operational state and intent visible.

    Ingestion Workflows

    Ingestion workflows convert source material into architecture work candidates or governed knowledge inputs.

    Current examples:

    • conversation-to-architecture-work exists as a step and prompt/workflow asset but is not active in the current default flow registry.
    • vision-to-architecture-work and discovery-to-architecture-work exist as workflow documentation and represent future or adjacent onboarding candidates.

    These workflows should be onboarded carefully because they affect input ownership and architecture work creation.

    Publication Workflows

    Publication workflows prepare governed knowledge for external consumption, dashboards, portals, or delivery surfaces.

    Current related surfaces include dashboard, steering projection, and portal publication work. These workflows should preserve the distinction between generating governed knowledge and rendering or publishing it.

    Workflow Onboarding Process

    A workflow becomes a first-class CAS capability through the following onboarding process.

    1. Establish Workflow Purpose

    Define:

    • workflow name
    • lifecycle phase
    • governance participation role
    • category
    • source inputs
    • produced outputs
    • human decision boundaries

    The workflow must be tied to a CAS lifecycle phase or governance participation loop.

    2. Define Responsibility Boundary

    Clarify what belongs to:

    • CAS Runner
    • workflow step
    • prompt
    • schema
    • Knowledge Provider
    • human governance

    The workflow must not blur orchestration, reasoning, and persistence responsibilities.

    3. Define Input Contract

    List all required inputs and identify how each is supplied:

    • runner context
    • Knowledge Provider
    • cas-config.yaml
    • prompt metadata
    • previous step output
    • human-provided source material

    Inputs must not be implied through undocumented filesystem assumptions.

    4. Define Output Contract

    List top-level output keys and governed artifact destinations.

    Before runtime onboarding, confirm whether persist_outputs supports each output type. If not, provider support must be added before the workflow is treated as first-class.

    5. Add Step Implementation

    Create or update a step object with the standard lifecycle:

    • name
    • prepare_inputs
    • execute
    • validate

    The step should keep deterministic transformations explicit and call semantic reasoning only through the Reasoning Execution Contract.

    6. Add Prompt Contract

    Add prompt files under the configured prompt base path using the flow and step naming convention.

    Where possible, include frontmatter for:

    • schema
    • semantic contract version
    • prompt version

    7. Add Schema Contract

    Add or reference JSON schemas for structured outputs.

    Schema validation should be part of onboarding for artifacts that downstream steps or persistence logic consume.

    8. Register Flow or Step

    Register the step in the step registry and add it to the appropriate flow order.

    If a step is implemented but intentionally inactive, document that status explicitly. Implemented but unregistered steps are not first-class runtime workflow capabilities.

    9. Validate Execution

    Validation should cover:

    • smallest relevant step tests
    • flow-level runner behavior
    • schema validation
    • persistence behavior
    • telemetry behavior where reasoning is used
    • dry-run behavior if supported

    10. Update Governed Knowledge

    Update affected models, workflow mappings, diagrams, prompts, rules, and steering guidance where the onboarding changes platform behavior.

    Current Gaps and Recommendations

    The current implementation already establishes the core execution model, but several gaps should be made explicit.

    Existing Behavior

    • The runner uses a fixed flow registry.
    • Steps expose a consistent lifecycle shape.
    • Prompt discovery is convention-based.
    • Reasoning is provider-abstracted and schema-aware.
    • Persistence is centralized through the Knowledge Provider.
    • Telemetry and evaluation are part of non-dry-run execution.
    • Some workflows exist as documentation or code assets without being active runtime capabilities.

    Gaps

    • Flow dependencies are implicit in ordered lists rather than described as a declarative dependency graph.
    • Step output keys are validated locally but not governed by a unified workflow contract registry.
    • Persistence support is output-key based and must be manually aligned with new workflow artifacts.
    • Prompt schema mappings are partly hard-coded by step name and partly available through prompt metadata.
    • Implemented but inactive steps can be mistaken for onboarded runtime capabilities.
    • The workflow category model is conceptual and not yet represented in runtime metadata.

    Recommendations

    • Introduce a Workflow Manifest pattern when the number of runtime workflows makes implicit registry entries difficult to govern.

    A Workflow Manifest would provide a machine-readable representation of workflow metadata while preserving the current execution model.

    Conceptual example:

    workflow:
  name: conversation-to-architecture-work
  category: ingestion
  lifecycle_phase: baseline_discovery

inputs:
  - conversations

outputs:
  - architecture_work

prompts:
  - conversation-to-architecture-work.v1.md

schemas:
  - architecture-work.schema.json
    • Require each newly onboarded workflow to document input keys, output keys, persistence destinations, prompt path, schema path, and lifecycle phase.
    • Prefer prompt frontmatter schema references over hard-coded schema mappings for new workflows.
    • Make inactive or experimental steps explicit in registry documentation.
    • Keep orchestration deterministic unless a future workflow has a documented need for branching or dynamic dependency resolution.

    These are recommendations for future consistency, not prerequisites for the current runner to remain valid.

    Relationship to Existing Models

    Relationship to the CAS Project Lifecycle Model

    The Project Lifecycle Model explains how projects move through Bootstrap, Baseline Discovery, Governance Establishment, Continuous Governance, and Rehydration.

    The Workflow Execution Model explains how executable workflows support those phases.

    Examples:

    • Bootstrap prepares the substrate and workspace required for execution.
    • Discovery workflows establish baseline understanding.
    • Governance workflows support Governance Establishment and Continuous Governance.
    • Rehydration workflows would restore executable understanding when knowledge state becomes incomplete.

    Relationship to the CAS Operational Governance Model

    The Operational Governance Model explains the strategic governance philosophy of Prevent, Detect, Feedback, operational intelligence, trust boundaries, and human governance.

    The Workflow Execution Model explains how those governance loops become executable through runner-managed pipelines, workflow steps, prompts, schemas, and knowledge persistence.

    Relationship to the CAS Governance Workflow Mapping

    The Governance Workflow Mapping explains how workflows participate in governance loops.

    The Workflow Execution Model defines the runtime contract required for those workflows to execute consistently inside CAS Runner.

    The mapping answers where a workflow participates. This model answers how a workflow is onboarded and executed.

    Relationship to the CAS Project Bootstrap Model

    The Bootstrap Model distinguishes the Governance Substrate from the Knowledge Workspace.

    The Workflow Execution Model depends on both:

    • the substrate provides agent instructions, governance rules, configuration, validation, and execution participation assets
    • the workspace provides the persisted knowledge that workflows load and update

    Bootstrap enables participation. Workflow execution operationalizes participation.

    Relationship to the Knowledge Provider Model

    The Knowledge Provider boundary is the execution model's persistence and retrieval contract.

    The runner and steps should treat governed knowledge as provider-supplied, not as arbitrary filesystem state. This supports local filesystem execution today while preserving the possibility of future provider implementations.

    Architectural Principles

    Separation of Concerns

    Orchestration, reasoning, persistence, prompt definition, schema validation, and human governance are distinct responsibilities.

    Knowledge First

    Workflows operate against governed knowledge inputs and persist governed outputs. Execution should strengthen the Knowledge Layer rather than create untracked side effects.

    Runtime Orchestration

    CAS workflows are executed by a deterministic runtime pipeline. The runner coordinates; workflow steps transform.

    Governance by Design

    Rules, prompts, schemas, provider boundaries, and persistence destinations are part of the workflow contract, not after-the-fact checks.

    Continuous Governance

    Every workflow execution can contribute to updated state, alignment findings, architecture work, intent, telemetry, or operational intelligence.

    Contract-First Architecture

    New workflows should become first-class through explicit contracts for inputs, outputs, prompts, schemas, dependencies, and persistence.

    Platform Contract Summary

    A CAS workflow is first-class when:

    • it has a registered runtime identity
    • it participates in a known lifecycle or governance loop
    • it executes through CAS Runner or an equivalent governed runner surface
    • its steps implement the standard step lifecycle
    • its prompts are discoverable through the Prompt Repository
    • its semantic reasoning goes through the Reasoning Execution Contract
    • its structured outputs are validated
    • its governed outputs can be persisted through the Knowledge Provider
    • its dependencies are explicit
    • its execution is observable
    • its human governance boundaries are clear

    This model establishes the authoritative execution contract for onboarding future workflows into CAS Runner while preserving the current architecture: runner-managed orchestration, workflow-owned reasoning, provider-owned knowledge, and governance-aware persistence.
